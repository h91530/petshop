# 펫샵 쇼핑몰 (Petshop) — 포트폴리오 정리

## 1. 프로젝트 개요

- **한 줄 소개**: 반려동물 용품을 사고파는 이커머스 사이트. 상품 조회/찜/장바구니/주문/결제(토스페이먼츠)/리뷰/반품까지 실제 쇼핑몰의 전체 구매 흐름을 구현.
- **배포**
  - 프론트엔드: `https://pet.yangtae.site` (Vercel)
  - 백엔드 API: `https://api.yangtae.site`
  - GitHub: `github.com/h91530/petshop`
- **개발 형태**: 프론트엔드 단독 개발. 백엔드(Express + MySQL)는 별도로 구축된 API 서버와 연동.

## 2. 기술 스택

| 영역 | 기술 |
|---|---|
| 프레임워크 | Next.js 16 (App Router, Turbopack) |
| 언어 | TypeScript |
| 서버 상태 관리 | TanStack React Query (useQuery/useMutation, 캐시 무효화) |
| 클라이언트 상태 관리 | Zustand (인증/모달/토스트/컨펌 스토어) |
| 스타일 | 순수 CSS (컴포넌트/페이지 단위 분리, 반응형 미디어쿼리) |
| 결제 | 토스페이먼츠 SDK (`@tosspayments/tosspayments-sdk`) |
| 백엔드(연동) | Express + MySQL (JWT + httpOnly 쿠키 인증) |
| 배포 | Vercel (프론트), 별도 서버 (백엔드) |

## 3. 주요 기능

- **상품**: 목록/상세, 카테고리 필터(강아지/고양이), 옵션 선택, 평균 별점·리뷰 수 노출
- **찜(좋아요)**: 토글, 헤더 뱃지 카운트 실시간 반영
- **장바구니**: 담기/수량 변경/삭제(단건·일괄), 총액 계산
- **주문/결제**:
  - 장바구니 전체 주문 & 상품 상세에서 바로구매 두 가지 플로우
  - 토스페이먼츠 위젯 연동 (결제창 로딩 완료 후 한 번에 표시하도록 UX 개선)
  - 서버 사이드 금액 검증(클라이언트가 보낸 금액과 서버가 계산한 금액 대조 후 결제 승인)
  - 개발용 테스트 결제(토스 결제창을 거치지 않고 즉시 결제 완료 처리) — 운영 환경에서는 라우트 차단
- **주문내역**: 목록/상태 표시, 배송지 표시, 반품 신청(결제 후 7일 이내), 반품 신청 시 리뷰 작성 버튼 자동 차단
- **리뷰**: 작성(결제 후 3일 이내 + 미작성 + 반품 미신청 조건), 목록(최신순/별점순 정렬 + 페이지네이션), 삭제, 내가 쓴 리뷰 모아보기
- **회원**: 회원가입(주소 검색 — 다음/카카오 우편번호 API 연동), 로그인/로그아웃, 마이페이지(정보 조회/수정)
- **반응형**: 전 페이지 모바일 대응 — 헤더 햄버거 드로어, 카드/리스트 스택 전환, 입력 요소 iOS 대응 등

## 4. 아키텍처 특징

### 폴더 구조 컨벤션
```
app/
  data/     → fetch 함수 (백엔드 API 호출부, apiClient.ts 래퍼 사용)
  hooks/    → data 함수를 감싼 useQuery/useMutation 훅
  store/    → zustand 전역 상태 (auth, modal, toast, confirm)
  components/  → 재사용 UI 컴포넌트
  (페이지 디렉터리)/  → 라우트별 page.tsx + css
```
`data → hooks → components` 단방향 흐름으로 서버 통신 로직과 UI를 분리. 모든 백엔드 요청은 `app/lib/apiClient.ts`의 `apiFetch` 래퍼 하나를 거치도록 통일해, 백엔드 도메인이 바뀌어도 한 곳만 수정하면 되게 구성.

### 인증 방식
- JWT를 **httpOnly + Secure + SameSite=None 쿠키**로 관리 (프론트 JS에서 토큰에 직접 접근 불가 → XSS로부터 토큰 보호)
- 요청마다 쿠키를 슬라이딩 방식으로 재발급(백엔드에서 요청마다 새 만료시간의 토큰을 다시 내려줌)해 세션을 자동 연장
- 상태 확인은 `/me` 요청 성공 여부로 판단, react-query로 캐싱 및 무효화 관리

### 상태 관리 원칙
- 서버에서 온 데이터(상품, 장바구니, 주문, 리뷰 등)는 전부 React Query로 관리, mutation 성공 시 관련 쿼리 `invalidateQueries`로 동기화
- 클라이언트 전용 상태(로그인 여부, 모달 열림, 토스트 메시지)만 zustand 사용
- zustand는 항상 selector 문법(`useStore(s => s.field)`)으로 구독해 불필요한 리렌더링 방지

## 5. 트러블슈팅 (기술 면접 포인트)

### 5.1 전체 페이지 비반응형 → `.inner` 고정 폭 버그
**증상**: 모바일에서 레이아웃 전체가 잘리거나 가로 스크롤 발생.
**원인**: 공통 컨테이너 클래스 `.inner`가 `width: 1200px`로 고정되어 있었음(= max-width가 아니라 width).
**해결**: `width: 100%; max-width: 1200px; padding: 0 20px;`로 전역 수정. 이 한 줄이 사이트 전체 반응형의 근본 원인이었음 — 국소적 미디어쿼리보다 먼저 전역 레이아웃 기반을 점검해야 한다는 교훈.

### 5.2 iOS Safari 입력창 자동 확대
**증상**: 아이폰에서 입력창을 탭하면 화면이 확대됨.
**원인**: iOS Safari는 포커스된 input/select/textarea의 `font-size`가 16px 미만이면 자동으로 확대하는 사양이 있음. 모바일 반응형 작업 중 입력창 폰트를 14~15px로 줄인 게 원인.
**해결**: 사이트 내 모든 input/select/textarea의 font-size를 16px 이상으로 통일. 단, 버튼/라벨 등 텍스트 입력이 아닌 요소는 이 제약과 무관하므로 자유롭게 축소 가능하다는 걸 구분해서 적용.

### 5.3 iOS에서만 로그인이 풀리는 문제 (크로스사이트 쿠키 차단)
**증상**: 데스크톱에서는 로그인이 정상 동작하지만, 아이폰(iOS 계열 브라우저 전부 — WebKit 엔진 공유)에서는 로그인 요청은 성공하는데 로그인 상태가 유지되지 않음.
**원인 분석**: 프론트엔드 도메인과 백엔드 도메인이 서로 다른 등록 도메인(예: `yangtae.site` vs `yangti.shop`)이었음. `credentials:"include"`로 쿠키를 주고받는 구조였는데, iOS의 ITP(Intelligent Tracking Prevention)가 이런 크로스사이트 쿠키를 서드파티 트래킹으로 간주해 저장을 차단함. 로그인 API 응답(`Set-Cookie`)은 200으로 오지만 브라우저가 쿠키 저장 자체를 막아버려 다음 `/me` 요청에서 곧바로 로그아웃 상태로 보이는 것.
**해결 검토**: (1) Authorization 헤더 + 토큰 저장 방식으로 전환하는 방법도 검토했으나 XSS 노출 리스크로 보류. (2) **프론트/백엔드를 같은 등록 도메인의 서브도메인으로 통일**(`pet.yangtae.site` / `api.yangtae.site`)해 브라우저 입장에서 same-site 요청이 되도록 인프라를 변경 → ITP가 더 이상 차단하지 않음. 인프라 단으로 원인을 해결한 사례.

### 5.4 flex-basis는 주축(main-axis)을 따라간다
여러 군데에서 반복적으로 만난 버그: `flex: 1 1 0`을 준 요소가 `flex-direction: row`일 땐 너비를 채우지만, 같은 규칙이 `flex-direction: column`으로 바뀌면 오히려 **높이**를 제어하게 되어 명시적으로 준 `height`가 무시됨. 모바일 브레이크포인트에서 `flex-direction`을 바꿀 때마다 `flex: 0 0 auto`로 리셋해줘야 한다는 패턴을 확립해서 이후 버그를 예방.

### 5.5 z-index 스태킹 버그
모바일 헤더를 `position: sticky`로 바꾸면서 `z-index: 220`을 줬는데, 로그인/회원가입 모달의 `z-index`가 100으로 더 낮아서 모바일에서 모달이 헤더에 가려 잘려 보이는 문제 발생. 모달을 `z-index: 500`으로 올려 항상 최상단에 오도록 수정 — 반응형 작업 시 새로 추가한 sticky 요소의 z-index가 기존 오버레이 요소들과 충돌하지 않는지 함께 점검해야 함을 확인.

### 5.6 결제 금액 위변조 방지
결제 승인(`/payments/confirm`) 요청 시 클라이언트가 보낸 금액을 그대로 믿지 않고, 서버가 장바구니(혹은 바로구매 상품)를 기준으로 직접 재계산한 금액과 대조한 뒤에만 토스 결제 승인 API를 호출하도록 설계(백엔드 협업 포인트). 프론트는 UI/UX(로딩 완료 후 결제창 표시, 실패/취소 시 PENDING 주문 정리)에 집중.

## 6. 반응형 작업에서의 설계 원칙 (직접 정리한 패턴)

- 카드형 리스트는 `aspect-ratio`로 이미지 비율을 고정하고 `object-fit: cover`로 왜곡 방지
- Next.js `<Image>`를 쓰는 요소는 wrapper에 명시적 `width/height`를 줘야 미디어쿼리로 안정적으로 크기 재정의 가능(컴포넌트 props 크기에만 의존하면 반응형 오버라이드가 불안정함)
- 리스트 아이템은 "이미지+정보" 묶음과 "액션 버튼" 묶음을 별도 wrapper로 나눠야, 모바일에서 두 그룹의 stacking 방향을 독립적으로 제어할 수 있음
- 브레이크포인트는 페이지마다 콘텐츠 특성에 맞게 개별 설정(헤더 900px, 상품 상세 850px, 리스트류 768/700px 등) — 하나의 값으로 통일하지 않고 실제 콘텐츠가 깨지는 지점을 기준으로 결정
